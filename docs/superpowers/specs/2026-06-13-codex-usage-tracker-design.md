# Codex Usage Tracker Design

## Goal

Build a cross-platform, no-UI Codex token usage tracking system. Each machine runs a lightweight client that scans local Codex session logs every 5 minutes and reports token usage to a public server. The server stores events in SQLite, calculates cost, aggregates hourly and daily usage, and serves a web dashboard.

## Confirmed Decisions

- Collection approach: scan local Codex session JSONL logs.
- Client platforms: macOS and Windows.
- Client UI: none.
- Deployment target: public server or VPS.
- Language and runtime: Go for client, server, and embedded web assets.
- Database: SQLite.
- Identity rule: `identity_key` is explicitly configured on each client. Same value merges usage, different values split usage, even if the underlying Codex account is the same.
- Client configuration: local YAML file.
- Client runtime mode: support both foreground `run` and service installation for Windows Service and macOS launchd.

## Reference Findings

`D:\code\stock\CodexBar-main` contains CodexBar logic for Codex OAuth usage, local cost scanning, model pricing, and identity handling. It confirms that Codex auth data normally lives under `CODEX_HOME` or `~/.codex`, and that local session/cost scans need careful background execution.

`D:\code\stock\cc-switch-main` contains a directly relevant Codex session usage scanner in `src-tauri/src/services/session_usage_codex.rs`. The scanner reads `~/.codex/sessions/**/*.jsonl` and `~/.codex/archived_sessions/*.jsonl`, parses `session_meta`, `turn_context`, and `event_msg` records whose payload type is `token_count`, then derives per-request deltas from cumulative token usage. Its proxy usage code also confirms the OpenAI/Codex pricing semantic: `input_tokens` includes cache-read tokens, so fresh billable input is `max(input_tokens - cache_read_tokens, 0)`.

## Architecture

The system is a Go monorepo with three executable targets:

- `codex-usage-client`: no-UI macOS/Windows client. It reads config, generates or loads a stable `device_id`, scans Codex session logs incrementally, persists local scan state and an upload retry queue, and posts batches to the server.
- `codex-usage-server`: public HTTP service. It accepts authenticated event batches, writes them idempotently into SQLite, calculates cost from server-side model prices, updates rollups, and exposes query APIs.
- `codex-usage-web`: embedded static web dashboard served by `codex-usage-server`.

Data flow:

1. Codex writes local session JSONL logs.
2. The client scans new or changed JSONL lines every 5 minutes.
3. The client extracts token deltas and emits stable usage events.
4. The client sends events to `POST /api/v1/ingest` with a bearer API key.
5. The server validates the key, records device presence, calculates cost, writes new events by `event_id`, and updates hourly/daily rollups.
6. The dashboard queries summary, breakdown, and recent event APIs.

## Client Configuration

Default config locations:

- Windows: `%APPDATA%\codex-usage-client\config.yaml`
- macOS: `~/Library/Application Support/codex-usage-client/config.yaml`

Example:

```yaml
server_url: "https://your-domain.com"
api_key: "server-issued-key"
identity_key: "a"
scan_interval: "5m"
codex_home: ""
```

`codex_home` resolution:

1. Use `codex_home` if non-empty.
2. Else use `CODEX_HOME` if set.
3. Else use the platform home directory plus `.codex`.

Client local state:

- `device_id`: UUID generated by `codex-usage-client init` or first `run`.
- `scan_state`: per-file cursor data, including file path, platform file identity when available, last modified time, size, last read offset or line number, current session id, current model, previous cumulative token values, and event index.
- `pending_queue`: events that could not be uploaded because of network or server errors.

## Client Scanning

Scanned paths:

- `<codex_home>/sessions/**/*.jsonl`
- `<codex_home>/archived_sessions/*.jsonl`

Relevant record types:

- `session_meta`: extract `session_id` from `payload.session_id`, `payload.sessionId`, or `payload.id`.
- `turn_context`: extract current model from `payload.model` or `payload.info.model`.
- `event_msg` with `payload.type == "token_count"`: extract usage.

Usage extraction:

- Prefer `payload.info.total_token_usage` as cumulative usage.
- Fall back to `payload.info.last_token_usage` as an already incremental usage value.
- Read `input_tokens`, `cached_input_tokens` or `cache_read_input_tokens`, and `output_tokens`.
- Clamp `cache_read_tokens` so it never exceeds `input_tokens`.
- Skip all-zero deltas.

Delta rules:

- For cumulative usage: `delta = current_cumulative - previous_cumulative`.
- For last usage: `delta = last_usage`.
- Preserve previous cumulative state across scan cycles so incremental scans remain correct.

Model normalization:

- Lowercase model names.
- Strip provider prefixes such as `openai/`.
- Strip known date suffixes like `-2026-05-14` or `-20260514` when matching prices.
- Preserve the raw model in the event if useful for debugging; use normalized model for pricing lookup.

Event identity:

```text
sha256(device_id + file_identity + session_id + event_index + occurred_at)
```

The event id must remain stable across repeated scans of the same file on the same device. The server also enforces uniqueness on `event_id`.

Event payload shape:

```json
{
  "event_id": "sha256...",
  "device_id": "uuid",
  "identity_key": "a",
  "session_id": "session-id",
  "source": "codex_session",
  "model": "gpt-5.5",
  "input_tokens": 1234,
  "cache_read_tokens": 500,
  "cache_creation_tokens": 0,
  "output_tokens": 321,
  "occurred_at": "2026-06-13T10:15:00Z"
}
```

Cost is not calculated by the client. The client reports token facts only.

## Client Runtime

Commands:

```text
codex-usage-client init
codex-usage-client run
codex-usage-client install-service
codex-usage-client uninstall-service
```

`run` starts the foreground loop. `install-service` installs the same loop as:

- Windows Service on Windows.
- launchd agent on macOS.

The first implementation should make `run` reliable before relying on service installation. Service install commands should use the same config file and log to a predictable platform-specific log path.

## Server Configuration

Example:

```yaml
listen_addr: ":8080"
public_base_url: "https://your-domain.com"
api_key: "change-me"
sqlite_path: "./data/codex-usage.db"
```

SQLite startup settings:

```sql
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
PRAGMA foreign_keys=ON;
```

The service is single-process in the first version. SQLite is acceptable because event writes are small, batch-oriented, and idempotent.

## Server Data Model

```sql
CREATE TABLE devices (
  device_id TEXT PRIMARY KEY,
  identity_key TEXT NOT NULL,
  label TEXT,
  platform TEXT,
  first_seen_at TEXT NOT NULL,
  last_seen_at TEXT NOT NULL
);

CREATE TABLE usage_events (
  event_id TEXT PRIMARY KEY,
  device_id TEXT NOT NULL,
  identity_key TEXT NOT NULL,
  session_id TEXT,
  source TEXT NOT NULL,
  model TEXT NOT NULL,
  input_tokens INTEGER NOT NULL,
  cache_read_tokens INTEGER NOT NULL,
  cache_creation_tokens INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL,
  occurred_at TEXT NOT NULL,
  received_at TEXT NOT NULL,
  pricing_status TEXT NOT NULL,
  cost_usd REAL,
  FOREIGN KEY(device_id) REFERENCES devices(device_id)
);

CREATE TABLE model_prices (
  model TEXT PRIMARY KEY,
  input_per_million REAL NOT NULL,
  cache_read_per_million REAL NOT NULL DEFAULT 0,
  cache_creation_per_million REAL NOT NULL DEFAULT 0,
  output_per_million REAL NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE usage_rollups (
  bucket_start TEXT NOT NULL,
  bucket_granularity TEXT NOT NULL,
  identity_key TEXT NOT NULL,
  device_id TEXT NOT NULL,
  model TEXT NOT NULL,
  input_tokens INTEGER NOT NULL,
  cache_read_tokens INTEGER NOT NULL,
  cache_creation_tokens INTEGER NOT NULL,
  output_tokens INTEGER NOT NULL,
  total_tokens INTEGER NOT NULL,
  cost_usd REAL,
  event_count INTEGER NOT NULL,
  PRIMARY KEY(bucket_start, bucket_granularity, identity_key, device_id, model)
);
```

Additional indexes should cover:

- `usage_events(identity_key, occurred_at)`
- `usage_events(device_id, occurred_at)`
- `usage_events(model, occurred_at)`
- `usage_rollups(bucket_granularity, bucket_start)`
- `usage_rollups(identity_key, bucket_granularity, bucket_start)`

## Cost Calculation

The server calculates cost with server-side model prices. Codex/OpenAI token semantics treat `input_tokens` as including cache-read tokens, so the billable fresh input is:

```text
billable_input = max(input_tokens - cache_read_tokens, 0)
```

Cost:

```text
cost =
  billable_input * input_per_million / 1_000_000 +
  cache_read_tokens * cache_read_per_million / 1_000_000 +
  cache_creation_tokens * cache_creation_per_million / 1_000_000 +
  output_tokens * output_per_million / 1_000_000
```

If a model has no price row:

- Store the event.
- Set `pricing_status = "missing"`.
- Set `cost_usd = NULL`.
- Include the model in API/dashboard missing-pricing warnings.

This allows token usage to remain accurate while prices are added in a future pricing-maintenance step.

## Server API

Authentication:

```text
Authorization: Bearer <api_key>
```

First version uses one global server API key.

### `POST /api/v1/ingest`

Request:

```json
{
  "device_id": "uuid",
  "identity_key": "a",
  "platform": "windows",
  "events": []
}
```

Behavior:

- Validate bearer API key.
- Validate device id and identity key.
- Upsert `devices`.
- Insert events with `INSERT OR IGNORE` or equivalent idempotent handling.
- Calculate cost for new events.
- Update hourly and daily `usage_rollups`.
- Return accepted count, duplicate count, and missing-price models.

### `GET /api/v1/summary`

Query parameters:

- `from`
- `to`
- `identity_key`
- `granularity=hour|day`

Returns totals and time buckets for dashboard charts.

### `GET /api/v1/breakdown`

Query parameters:

- `from`
- `to`
- `group_by=identity|device|model`

Returns token and cost breakdowns.

### `GET /api/v1/events`

Query parameters:

- `from`
- `to`
- `limit`

Returns recent event details for debugging.

### `GET /healthz`

Returns service health and SQLite connectivity status.

## Web Dashboard

The server serves the dashboard at `/` using embedded static assets.

First-version views:

- Overview metrics: total cost, total tokens, event count, missing-price model count.
- Trend chart: token and cost by hour/day.
- Filters: time range, identity key, device, model.
- Breakdown table: group by identity, device, or model.
- Recent events table for debugging client ingestion.
- Missing-pricing list.

The dashboard should be operational and information-dense, not a marketing page.

## Identity Semantics

`identity_key` is the only merge key for usage identity.

Examples:

- Computer 1 uses Codex account A and configures `identity_key: a`.
- Computer 2 uses the same Codex account A and configures `identity_key: b`.
- Usage is split into `a` and `b`.
- If Computer 2 also configures `identity_key: a`, usage is merged into `a`.

The real Codex account email or account id may be added as optional metadata in a future version, but it must not override the configured identity behavior.

## Error Handling

Client:

- Network failure: keep events in `pending_queue` and retry on the next cycle.
- Server 401/403: stop uploading, log authentication failure, retain pending events.
- Malformed JSONL line: skip line and continue.
- Missing Codex sessions directory: report a local status warning, keep running.
- Clock skew: send `occurred_at`; server also stores `received_at`.

Server:

- Duplicate event id: count as duplicate, not error.
- Missing model price: store event with `pricing_status = "missing"`.
- Invalid token counts: reject event with validation error.
- SQLite busy: rely on `busy_timeout`; return 503 if still blocked.

## Testing Strategy

Client unit tests:

- Codex JSONL token-count parsing.
- Cumulative token to delta conversion.
- `last_token_usage` fallback.
- Repeated scans do not duplicate events.
- Pending queue survives upload failure.
- macOS and Windows config/path resolution.

Server unit/integration tests:

- `/api/v1/ingest` authentication.
- Idempotent event insert by `event_id`.
- Cost calculation with cache-inclusive input semantics.
- Missing model price handling.
- Hourly and daily rollup updates.
- Summary and breakdown queries.

End-to-end fixture:

- Device 1 with `identity_key = a`.
- Device 2 with `identity_key = b`.
- Device 3 with `identity_key = a`.

Expected result:

- `a` combines devices 1 and 3.
- `b` remains separate.
- Totals by model and day match fixture token deltas.

## Out of Scope For First Version

- Client graphical UI.
- Automatic identity merge from OpenAI/Codex account email.
- Multi-tenant user accounts.
- Per-device API tokens.
- Automatic public model pricing refresh.
- Proxy-based realtime usage capture.
- PostgreSQL support.

These are extension points after the first version is stable.
